========================================================================
ROMA · Punti di interesse + meteo
Roadmap delle modifiche e limiti del progetto
========================================================================
Ultimo aggiornamento: 25/06/2026 (rev. layout mobile)

------------------------------------------------------------------------
1. COS'È OGGI IL PROGETTO
------------------------------------------------------------------------
Single-page HTML (index.html) servita da nginx (Dockerfile) e deployata
su Cloud Run (cloudbuild.yaml). Nessun backend applicativo: tutta la
logica è lato browser.

L'app è un VISUALIZZATORE DI PUNTI DI INTERESSE di Roma, con il meteo
generale della città. Funzioni attuali:
- Mappa Leaflet su sfondo OpenStreetMap centrata su Roma.
- Riquadro METEO DI ROMA in alto a destra: condizione (emoji sole/
  pioggia/nuvole/temporale dal weather_code WMO + giorno/notte),
  temperatura, percepito, umidità, vento. Dati live da Open-Meteo,
  auto-refresh ogni 10 minuti con orario di aggiornamento.
- Menu in alto a sinistra per scegliere il punto di interesse:
    * Nasoni / fontanelle (⛲, amenity=drinking_water da OSM)
    * Parchi oltre 3000 m² (🌳, leisure=park da OSM)
  caricati live via Overpass entro un raggio regolabile (slider 0,5-5 km)
  dalla posizione GPS dell'utente.
- Geolocalizzazione (GPS) con marker "sei qui" e cerchio del raggio;
  fallback al centro di Roma quando il GPS non è disponibile (test).
- Clustering dei marker, popup informativi sui POI.
- Robustezza: fallback su più mirror Overpass, banner di errore con
  "Riprova", info-box collassabile e layout adattato su mobile.
- Layout MOBILE (schermi <=480px) ottimizzato:
    * Riquadro meteo/temperatura compatto e in VERTICALE (le voci
      Percepiti/Umidità/Vento incolonnate invece che affiancate).
    * Pannello "Punti di interesse" spostato in BASSO a sinistra, che si
      apre verso l'ALTO al tocco della freccia (flex column-reverse).

Dipendenze esterne: Leaflet + Leaflet.markercluster (CDN unpkg),
API Open-Meteo (meteo), API Overpass (POI da OpenStreetMap),
tile di OpenStreetMap, Geolocation API del browser.


------------------------------------------------------------------------
2. STORIA DEL PROGETTO (contesto)
------------------------------------------------------------------------
Il progetto è nato come "mappa termica del microclima" con 24 zone di
Roma colorate per temperatura percepita e un menu di variabili meteo
(percepito/temperatura/vento/umidità). Sono state implementate e poi
ANNULLATE: una heatmap interpolata (Leaflet.heat) e una griglia
automatica di punti.

Successivamente l'app è stata RICONVERTITA in visualizzatore di punti di
interesse: rimosse le zone colorate e il meteo per-zona, sostituiti dal
riquadro meteo generale di Roma. I POI (nasoni/parchi) e la
geolocalizzazione, introdotti nella fase precedente, sono il cuore
attuale dell'app.


------------------------------------------------------------------------
3. LIMITI ATTUALI DEL PROGETTO
------------------------------------------------------------------------

DATI
- Il meteo del riquadro è quello del CENTRO di Roma (un solo punto),
  non della posizione reale dell'utente.
- Solo due tipi di POI (nasoni e parchi). Mancano molte categorie utili
  (aree picnic, bagni pubblici, panchine, punti panoramici, ecc.).
- L'area dei parchi è approssimata (anello per le "way", riquadro per le
  "relation"): il filtro 3000 m² è quindi indicativo.

DIPENDENZA DA SERVIZI ESTERNI
- Overpass pubblico è lento e a volte sovraccarico (risponde HTML al
  posto del JSON, errori 429/406/504). Mitigato con fallback su più
  mirror e rilevamento risposte non-JSON, ma NON è garantito: senza un
  backend di cache l'esperienza dipende dallo stato dei server pubblici.
- Nessuna cache: ogni cambio di raggio/POI rifà la query da zero.
- CDN unpkg per Leaflet/markercluster senza fallback: se unpkg è
  irraggiungibile la mappa non si carica (schermata bianca).

GEOLOCALIZZAZIONE
- Il GPS del browser richiede HTTPS: funziona su Cloud Run, NON in
  locale via http/file (lì scatta il fallback "centro di Roma").
- Se l'utente nega il permesso, i POI sono centrati sul centro città.
- Manca un pulsante per scegliere manualmente un punto sulla mappa.

ACCESSIBILITÀ / UX
- Nessuna etichetta ARIA, nessuna localizzazione (solo italiano).
- Il menu non ha un'icona/feedback di "caricamento in corso" oltre al
  testo nella legenda.

SICUREZZA / DEPLOY
- Cloud Run con --allow-unauthenticated, nessun rate limiting né CDN
  davanti (costi/abusi possibili).
- Nessun ambiente di staging; tag immagine basato solo su COMMIT_SHA.
- Nessun test automatico né linting (solo build Docker).


------------------------------------------------------------------------
4. MODIFICHE CONSIGLIATE (residue, dalla più utile in giù)
------------------------------------------------------------------------

FUNZIONALITÀ
[ ] Meteo sulla posizione dell'utente (oltre/al posto del centro Roma).
[ ] Più categorie di POI nel menu: aree picnic (tourism=picnic_site,
    leisure=picnic_table), bagni pubblici (amenity=toilets), panchine,
    punti panoramici, fontane monumentali, ecc., ognuna con la sua emoji.
[ ] Pulsante "scegli un punto sulla mappa" e "ricentra sul GPS"
    (utile anche in locale senza HTTPS).
[ ] Previsione oraria nel riquadro meteo (Open-Meteo hourly).
[ ] Mostrare la distanza del POI dalla posizione e/o ordinarli per
    vicinanza; link "apri in Google/Apple Maps" per le indicazioni.

ROBUSTEZZA / INFRASTRUTTURA
[ ] Backend leggero (Cloud Run / Cloud Functions) come proxy + cache
    verso Open-Meteo e Overpass: meno chiamate, risposte più veloci e
    affidabili, possibilità di rate limiting.
[ ] Self-host di Leaflet/markercluster (o fallback) per non dipendere
    da unpkg.
[ ] Cache lato client (es. sessionStorage) dei risultati Overpass per
    raggio/posizione, per non rifare la stessa query.

ACCESSIBILITÀ / QUALITÀ
[ ] Etichette ARIA e palette accessibile.
[ ] PWA installabile (offline-friendly per le tile e l'ultima risposta).
[ ] Test automatici minimi e linting in CI; ambiente di staging.


------------------------------------------------------------------------
5. PRIORITÀ CONSIGLIATA
------------------------------------------------------------------------
1) Più categorie di POI + meteo sulla posizione utente: è ciò che dà
   più valore alla nuova identità "punti di interesse".
2) Backend proxy + cache (Open-Meteo/Overpass): risolve l'affidabilità,
   oggi il punto più fragile.
3) Distanza/indicazioni verso i POI e previsione oraria.
